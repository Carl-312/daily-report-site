Tavily 在最新灰度里的位置

Tavily 当前不是 main 生产流水线的默认新闻源。它在 main.py run 内部只承担一个可开关的 post-fetch enrichment 阶段：在已有 source 抓取和去重之后，验证部分候选是否仍在 24 小时窗口内，并在数量不足时按受控域名补量。灰度 workflow 跑的是同一个 main.py run，但强制带上 --offline --enrichment on，只用于观察和产出证据。

main 生产/发布路径

默认/定时 不启用 Tavily

不传 --enrichment on，TAVILY_API_KEY 也不会被注入。main.py 会进入 enrichment gate，但结果是 disabled，主流程照常继续。

手动灰度 生产 workflow 内临时启用

维护者手动设置 enable_tavily=true 时，生产 workflow 会把 Tavily 打开，但是否发布仍由 publish 控制。这是“同一条生产流水线上的显式灰度”。

同一个 main.py run 内部，Tavily 插在哪里

1. fetch_all

从 configured sources 抓取原始新闻。Tavily 不在 sources/ 里。

2. dedupe

先对 source 结果去重，得到 Tavily 的输入候选。

3. enrichment gate

auto/on/off 在这里生效。默认配置是 off。

4. Tavily active

启用时执行 prefilter、verify、priority refill、secondary refill；official fallback 默认关闭。

5. save_json

保存最终 articles，并把 enrichment 诊断写进 JSON。

6. summarize

摘要只消费 JSON 中的最终文章，不会再次调用 Tavily。

7. build

build.py 把 content/ 生成 dist/。

Tavily 在最新灰度中承担的具体作用

不是主 source

它不替代 aibase、techcrunch、theverge。source 为空时可以尝试补量，但这不证明 source 层可以废弃。

是验证器

对去重后的候选做 exact verify：匹配 URL 或同域高相似标题，并要求 published_date 落在 24 小时窗口内。

是受控补量器

不足 min_articles=10 时按域名分层补量。当前 Gray Test 2 证明 secondary refill 能进入，但仍没有证明默认开启安全。

Gray Test 2 说明了什么

观察点 最新灰度事实 工程解释
入口 python3 main.py run --offline --enrichment on 灰度没有绕过业务入口，而是强制打开同一个 enrichment gate。
预算 reserved_refill_calls=2，verify_budget=5，total_calls=7 修复点是让 verify 不再吃掉全部预算，给 priority 和 secondary refill 留机会。
阶段结果 verify 接受 3 条，priority 接受 0 条，secondary 接受 5 条 Tavily 的有效价值目前主要出现在 secondary refill；priority 存在 metadata 质量问题。
最终产出 final_count=8，低于 min_articles=10 这证明灰度链路和预算修复有效，但不能证明 Tavily 可以默认开启。
决策 继续 enrichment.enabled=false 下一步应该先研究 priority refill 的 published_date 缺失，而不是直接扩大默认预算或开启 fallback。

读这张图时先看两条外层 workflow：main 生产线默认绕过 Tavily，gray 观察线强制打开 Tavily。再看内部 main.py run：Tavily 只位于 dedupe 和 save_json 之间。

Gray Test 3 小实验

域名分层

把 reuters.com、arstechnica.com、techcrunch.com 放到 priority refill，把 thenextweb.com 和 venturebeat.com 降到 secondary。

时间窗

只在 gray workflow 中把 strict_hours 临时放宽到 30，验证北京时间晚间运行造成的 24h 边界误杀。

单次结果深度

只在 gray workflow 中把 refill_max_results 临时提高到 12。Tavily 当前允许 max_results 到 20，本轮不改变生产默认值。

验收看 priority + secondary 的 accepted_count / result_count 是否接近 50%，以及 final_count 是否稳定高于 Gray Test 2 之后的 5-9 区间；不以默认开启 Tavily 作为本轮结论。

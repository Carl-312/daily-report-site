## Interpretation

This run validates the budget-conflict fix in a live runner:

- `reserved_refill_calls=2` held back enough budget for staged refill.
- verify was capped at `verify_budget=5` instead of consuming all 6 configured
  verify calls.
- secondary refill executed, which did not happen in old run `25680995172`.
- secondary refill accepted `5` candidates, raising the output from `3` verified
  source articles to `8` final articles.

This is a real improvement over the old budget-conflict sample, but it is still
not enough to enable Tavily by default. The run remained below `min_articles=10`
and exhausted the default `max_total_calls=7` budget.

## Limiting Factors

The scorecard primary limiter is `budget_exhausted`.

Contributing factors:

- Priority refill returned `8` results but accepted `0`.
- Priority refill `published_date_missing_rate=1.0`.
- Secondary refill worked (`5/5` accepted) but only had one call available.
- Official fallback remained disabled, as intended.

This means the next useful strategy question is not "does secondary refill ever
work?" It does. The next question is whether priority refill can be made useful
without weakening the 24-hour metadata gate, or whether budget/official fallback
experiments are justified as explicit gray settings.

## Decision

Do not enable Tavily by default.

Keep Tavily behind explicit gray controls. Treat this run as evidence that:

- The workflow now uploads normalized scorecards.
- The refill budget reservation works in Actions.
- The current priority refill domain/query combination still has a metadata
  quality problem.

## Cannot Prove

- This single run does not prove Tavily stability across days.
- It does not prove the default budget is sufficient to reach `10` final
  articles.
- It does not prove official fallback should be enabled.
- It does not prove broadening domains is safe.
